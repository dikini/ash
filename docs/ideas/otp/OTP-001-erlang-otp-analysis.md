## 1. Executive Summary

This document provides a comprehensive analysis of Erlang/OTP's `gen_server` and `supervisor` behaviors, examining their architecture, inter-process protocols, and implementation details. The goal is to assess feasibility and design considerations for implementing similar functionality within the Ash language ecosystem.

**Key Finding:** The mapping between Erlang/OTP and Ash is stronger than initially assessed. Both systems share:
- Process/workflow state isolation
- Message passing as the coordination mechanism
- Address-based process identification

**Critical Uncertainty:** The generic/concrete split of OTP behaviorsвҖ”how to separate generic framework code from user-specific callbacks without type classesвҖ”remains an open architectural question.

## 2. Erlang/OTP Foundation Concepts

### 2.1 The Actor Model in Erlang

Erlang implements the actor model at the language runtime level:

| Concept | Erlang Implementation | Ash Equivalent |
|---------|----------------------|----------------|
| Process | Lightweight VM thread (preemptively scheduled) | Workflow with isolated state |
| Process ID | `pid()` - opaque process identifier | Workflow address (typed by capability) |
| Mailbox | Per-process message queue | Channel capability |
| Pattern Matching | `receive` clause selection | Match on channel receive |
| Process Spawning | `spawn/1`, `spawn_link/1` | Workflow spawn |
| Links | Bidirectional process relationships | **Uncertain** - capability revocation? |
| Monitors | Unidirectional process observation | **Uncertain** - event streams? |

### 2.2 OTP Design Principles

OTP (Open Telecom Platform) provides a framework for structuring Erlang applications through **behaviors**вҖ”formalized patterns with callback modules:

1. **gen_server**: Generic client-server behavior
2. **gen_statem**: Generic state machine behavior
3. **gen_event**: Generic event handling behavior
4. **supervisor**: Process supervision and fault tolerance

These behaviors separate **generic code** (provided by OTP) from **specific code** (implemented by the developer via callbacks).

**Key Architectural Question:** How can Ash implement this generic/concrete split? See Section 9.

### 3.2 Message Protocol

#### 3.2.1 Synchronous Call Protocol

**Client Side:**
```erlang
gen_server:call(ServerRef, Request) -> Reply
gen_server:call(ServerRef, Request, Timeout) -> Reply
```

**Protocol Flow:**
1. Client generates unique reference: `make_ref()`
2. Client sends message: `{'$gen_call', {ClientPid, Ref}, Request}`
3. Client blocks waiting for: `{Ref, Reply}` or timeout

**Server Side (in gen_server main loop):**
```erlang
handle_call(Request, From, State) ->
    {reply, Reply, NewState} |
    {noreply, NewState} |
    {stop, Reason, Reply, NewState} |
    {stop, Reason, NewState}
```

**Key Parameters:**
- `From`: `{pid(), Tag}` - identifies the caller for reply
- Tag is a unique reference generated by the caller

**Reply Mechanism:**
```erlang
% Immediate reply (via return value)
{reply, Reply, NewState}

% Delayed reply (via explicit call)
gen_server:reply(From, Reply)
```

**Ash Equivalent:**
```ash
// Effect-based request/response
let response = effect Call {
    target: server_addr,
    request: payload,
    timeout: duration
};
```

#### 3.2.2 Asynchronous Cast Protocol

**Client Side:**
```erlang
gen_server:cast(ServerRef, Request) -> ok
```

**Protocol Flow:**
1. Client sends message: `{'$gen_cast', Request}`
2. Client immediately continues (non-blocking)

**Server Side:**
```erlang
handle_cast(Request, State) ->
    {noreply, NewState} |
    {stop, Reason, NewState}
```

**Characteristics:**
- No response possible
- Fire-and-forget semantics
- Used when client doesn't need confirmation

**Ash Equivalent:**
```ash
// Fire-and-forget via send effect
send chan_cap with msg;
```

#### 3.2.3 Info Messages (Non-OTP Messages)

```erlang
handle_info(Info, State) ->
    {noreply, NewState} |
    {reply, Reply, NewState} |
    {stop, Reason, NewState}
```

Handles messages not recognized as calls or casts (e.g., timeouts, monitoring signals, direct `!` sends).

### 3.3 Callback Specifications

```erlang
% Mandatory callbacks
-callback init(Args :: term()) ->
    {ok, State :: term()} |
    {ok, State :: term(), timeout() | hibernate | {continue, term()}} |
    {stop, Reason :: term()} |
    ignore.

-callback handle_call(Request :: term(), From :: {pid(), Tag :: term()}, State :: term()) ->
    {reply, Reply :: term(), NewState :: term()} |
    {reply, Reply :: term(), NewState :: term(), timeout() | hibernate | {continue, term()}} |
    {noreply, NewState :: term()} |
    {noreply, NewState :: term(), timeout() | hibernate | {continue, term()}} |
    {stop, Reason :: term(), Reply :: term(), NewState :: term()} |
    {stop, Reason :: term(), NewState :: term()}.

-callback handle_cast(Request :: term(), State :: term()) ->
    {noreply, NewState :: term()} |
    {noreply, NewState :: term(), timeout() | hibernate | {continue, term()}} |
    {stop, Reason :: term(), NewState :: term()}.

-callback handle_info(Info :: term(), State :: term()) ->
    {noreply, NewState :: term()} |
    {noreply, NewState :: term(), timeout() | hibernate | {continue, term()}} |
    {stop, Reason :: term(), NewState :: term()}.

% Optional callbacks
-callback terminate(Reason :: term(), State :: term()) -> term().
-callback code_change(OldVsn :: term(), State :: term(), Extra :: term()) ->
    {ok, NewState :: term()} |
    {error, Reason :: term()}.
```

### 3.4 State Management

**State Lifecycle:**
1. `init/1` creates initial state
2. State passed to each callback
3. Callback returns potentially modified state
4. On crash: state lost (unless recovered via supervisor restart)

**State Isolation:**
- Each gen_server has its own isolated state
- No shared memory between processes
- State accessible only within callback context

**Ash Mapping:** Workflows have isolated state. The workflow's state is not shared with other workflowsвҖ”this is equivalent to Erlang's process isolation.

### 3.5 Process Registration

```erlang
% Local registration (name visible on current node)
gen_server:start_link({local, Name}, Module, Args, Options)

% Global registration (name visible cluster-wide)
gen_server:start_link({global, Name}, Module, Args, Options)

% Via module (custom registry)
gen_server:start_link({via, Module, Name}, Module, Args, Options)
```

**ServerRef can be:**
- `pid()`
- `Name` (atom) - locally registered
- `{Name, Node}` - globally registered
- `{global, Name}` - global registry
- `{via, Module, Name}` - custom registry

### 3.6 Timeout and Hibernation

**Timeout Option:**
```erlang
{ok, State, Timeout}  % Timeout in milliseconds
```

When timeout expires without messages: `handle_info(timeout, State)` called.

**Hibernation:**
```erlang
{reply, Reply, State, hibernate}
```

Process reduces memory footprint until next message arrives.

### 3.7 Internal Implementation Structure

The gen_server.erl implementation uses a classic message loop:

```erlang
% Simplified structure
loop(Parent, Name, State, Mod, Time, HibernateAfterTimeout) ->
    receive
        {'$gen_call', From, Request} ->
            handle_call(From, Request, ...);
        {'$gen_cast', Request} ->
            handle_cast(Request, ...);
        {'EXIT', Parent, Reason} ->
            terminate(Reason, ...);
        Msg ->
            handle_info(Msg, ...)
    after Timeout ->
        handle_info(timeout, ...)
    end.
```

### 4.2 Restart Strategies

#### 4.2.1 one_for_one

**Behavior:** When a child dies, only that child is restarted.

```

```

**Use case:** Processes with startup dependencies (e.g., B depends on A, C depends on B).

#### 4.2.4 simple_one_for_one

**Behavior:** Special case of one_for_one for dynamic child addition.

**Characteristics:**
- All children are instances of the same module
- Children are added dynamically via `supervisor:start_child/2`
- No static child specifications
- Used for connection handlers, worker pools, etc.

### 4.3 Maximum Restart Intensity

Prevents infinite restart loops ("flapping"):

```erlang
{MaxR, MaxT}
```

- **MaxR**: Maximum number of restarts allowed
- **MaxT**: Time window in seconds

**Example:** `{5, 10}` means maximum 5 restarts in 10 seconds.

**Behavior:** If restart intensity exceeded, supervisor terminates itself with reason `shutdown`, triggering its own supervisor to take action.

### 4.4 Child Specification

```erlang
#{
    id => ChildId,           % Unique identifier (term())
    start => {M, F, A},      % {Module, Function, Args} - MFArgs
    restart => Restart,      % permanent | transient | temporary
    shutdown => Shutdown,    % brutal_kill | timeout() | infinity
    type => Type,            % worker | supervisor
    modules => Modules       % [module()] | dynamic
}
```

#### 4.4.1 Restart Types

| Type | Description |
|------|-------------|
| `permanent` | Always restarted, regardless of exit reason |
| `transient` | Restarted only if abnormal exit (not normal/shutdown) |
| `temporary` | Never restarted |

#### 4.4.2 Shutdown Values

| Value | Description |
|-------|-------------|
| `brutal_kill` | Unconditionally terminate with `exit(Child, kill)` |
| `Timeout` | Send shutdown signal, wait Timeout ms, then kill |
| `infinity` | Wait indefinitely for graceful shutdown |

### 4.5 Supervisor Startup Sequence

```erlang
supervisor:start_link(Module, Args) ->
    % 1. Call Module:init(Args)
    % 2. Validate child specifications
    % 3. Start children in order
    % 4. Link to each child
    % 5. Enter supervision loop
```

### 4.6 Supervision Tree Depth

OTP recommends:
- Maximum 7-10 levels in supervision tree
- Keep trees broad rather than deep
- Root supervisor typically starts application

## 5. Process Linking and Monitoring

### 5.1 Links

**Characteristics:**
- Bidirectional
- Automatic error propagation
- Used within supervision trees

```erlang
link(Pid)       % Create link
unlink(Pid)     % Remove link
spawn_link(Fun) % Spawn and link atomically
```

**Exit Signal Propagation:**
- When linked process crashes, exit signal sent to all linked processes
- Default behavior: receiving process also crashes (error propagation)
- Can be trapped: `process_flag(trap_exit, true)`

### 5.2 Monitors

**Characteristics:**
- Unidirectional (A monitors B, not vice versa)
- No error propagation
- Used for observation without coupling

```erlang
MonitorRef = monitor(process, Pid)  % Create monitor
demonitor(MonitorRef)               % Remove monitor
```

**Monitor Message:**
```erlang
{'DOWN', MonitorRef, process, Pid, Reason}
```

### 5.3 Comparison

| Feature | Links | Monitors |
|---------|-------|----------|
| Directionality | Bidirectional | Unidirectional |
| Error propagation | Yes (by default) | No |
| Use case | Supervision, tight coupling | Observation, loose coupling |
| Trapping | `trap_exit` flag | Not applicable |

## 6. Inter-Process Communication Details

### 6.1 Message Sending Primitives

```erlang
Pid ! Message                    % Send message (async)
Pid ! Msg1 ! Msg2                % Send multiple (right-to-left)

% Selective receive
receive
    Pattern1 -> Action1;
    Pattern2 -> Action2
after Timeout -> TimeoutAction
end.
```

### 6.2 Message Ordering Guarantees

Erlang provides **per-process ordering**:
- If process A sends M1 then M2 to process B, B receives M1 before M2
- No guarantee about ordering of messages from different senders
- No guarantee about delivery timing

### 6.3 Process Dictionary

```erlang
put(Key, Value)   % Store value
get(Key)          % Retrieve value
get()             % Get all values
erase(Key)        % Delete value
```

**Note:** Process dictionary is per-process mutable state. Generally discouraged but useful for specific cases (e.g., random seeds, transaction contexts).

## 7. Hot Code Loading

### 7.1 Code Change Mechanism

```erlang
code_change(OldVsn, State, Extra) -> {ok, NewState}
```

**Process:**
1. Load new module version
2. Old processes continue with old code
3. New processes use new code
4. `code_change/3` called to migrate state

### 7.2 Module Versions

Erlang VM maintains two versions of each module:
- **Current**: Used by new processes
- **Old**: Used by existing processes

When all processes using "old" version terminate, old code is purged.

## 8. Analysis: Mapping to Ash

### 8.1 Fundamental Similarities

| Aspect | Erlang/OTP | Ash |
|--------|-----------|-----|
| State isolation | Per-process | Per-workflow |
| Coordination | Message passing | Message passing via effects |
| Process identity | Pid | Workflow address |
| Concurrency | Lightweight processes | OS threads + green threads (planned) |

### 8.2 PIDs vs Workflow Addresses

| Capability | Erlang PID | Ash Address |
|------------|------------|-------------|
| Identity | Opaque identifier | Opaque address |
| Location transparency | `pid()` hides node location | Address may hide runtime location |
| Type of destination | Untyped (any message) | **Typed** - channel capability constrains protocol |
| Creation | `spawn()` returns pid | Workflow spawn returns address |
| Passing | Can be sent in messages | Can be passed via capabilities |
| Lifetime | Process death = invalid | Address may outlive process (depending on design) |

**Key Difference:** Erlang pids are *just* identifiers. An Ash address is a capability that carries protocol type information.

### 8.3 Message Passing as Effects

**Correction from initial analysis:** In Ash, message passing *is* effects. There is no separationвҖ”the `send` and `receive` workflow forms exercise channel capabilities.

| Erlang | Ash |
|--------|-----|
| `Pid ! Msg` | `send chan_cap with msg` |
| `receive` | `receive chan_cap` |
| Untyped, runtime errors | Typed, compile-time verification |

### 8.4 State Isolation

**Correction from initial analysis:** Ash workflows have isolated state, equivalent to Erlang processes.

```ash
// Each workflow has its own isolated state
workflow server(init: InitArgs) -> Result<(), Error> {
    // This state is not accessible from other workflows
    let state = initialize(init)?;
    
    loop {
        let msg = receive chan_cap;
        state = handle_message(state, msg)?;
    }
}
```

### 8.5 Thread Model

**Correction from initial analysis:** Ash uses a combined native/green thread model:
- OS threads for parallelism
- Tokio tasks for async concurrency
- Suspend/resume support planned for green-thread-like behavior

This is closer to Erlang's lightweight process model than initially characterized.

### 8.6 What Can Be Adapted

| OTP Feature | Adaptable? | Notes |
|-------------|------------|-------|
| Callback-based server | вҡ пёҸ **Uncertain** | Requires generic/concrete split solution (Section 9) |
| Synchronous calls | вң… Yes | Via async/await + effects |
| Asynchronous casts | вң… Yes | Fire-and-forget via send effect |
| State isolation | вң… Yes | Workflow isolation equivalent |
| Process registration | вң… Yes | Registry capability |
| Supervision trees | вҡ пёҸ **Uncertain** | Needs links/monitors equivalent |
| Restart strategies | вҡ пёҸ Partial | Can implement but semantics differ |
| Hot code loading | вқҢ No | Requires VM support |
| Links/monitors | вҡ пёҸ **Uncertain** | Needs design |
| **Typed message protocols** | вҡ пёҸ **Opportunity with constraints** | Session types (see besedarium in OTP-002) can improve on Erlang's untyped messages, but type-state encoding may be difficult in Ash |

**Key Opportunity:** Erlang's message passing is untypedвҖ”protocol violations are runtime errors. Ash's typed effects + session types could provide compile-time protocol verification.

**Constraint:** Session type implementations (e.g., besedarium) rely on type-state encoding where channel types carry protocol state (`EpChanSend<Msg, Cont>`). This approach is **difficult to implement in Ash** [reasons TBD]. Alternative encoding strategies may be needed.

### 8.7 Open Questions

1. **Links/Monitors**: How to model bidirectional error propagation in Ash? Is capability revocation sufficient?
2. **"Let it crash"**: Erlang embraces crashes; Ash uses `Result` types. How to reconcile?
3. **Supervision semantics**: Can restart strategies be expressed in the type system?

## 9. Architectural Uncertainty: The Generic/Concrete Split

### 9.1 The Problem

OTP behaviors achieve separation of concerns through a generic/concrete split:

```erlang
% Generic code (in gen_server.erl)
loop(State, Mod) ->
    receive
        {'$gen_call', From, Request} ->
            {reply, Reply, NewState} = Mod:handle_call(Request, From, State),
            reply(From, Reply),
            loop(NewState, Mod)
    end.

% Concrete code (user's callback module)
handle_call(get_count, _From, State) ->
    {reply, State#state.count, State}.
```

The generic code:
- Handles message routing
- Manages the message loop
- Implements timeout handling
- Provides process registration

The concrete code:
- Implements domain logic
- Manages specific state types
- Handles specific requests

### 9.2 How Erlang Achieves This

Erlang uses **dynamic dispatch** via module atoms:
```erlang
gen_server:start_link({local, ?MODULE}, ?MODULE, Args, [])
%                                     ^^^^^^^^
%                                     Callback module passed as atom
```

At runtime, `Mod:handle_call/3` resolves to the concrete implementation.

### 9.3 The Ash Challenge

Ash has **generics** but no **type classes** (traits with associated types/implicits). The question is: how to pass the concrete implementation to generic code?

---

**Option 1: Explicit Dictionary Passing (Ad-hoc Polymorphism)**

```ash
type GenServerOps<State, Request, Response> = {
    init: fn(InitArgs) -> Result<State, Error>,
    handle_call: fn(Request, State) -> Result<(Response, State), Error>,
    handle_cast: fn(Request, State) -> Result<State, Error>,
};

// Generic loop
fn gen_server_loop<State, Request, Response>(
    ops: GenServerOps<State, Request, Response>,
    state: State
) -> Result<(), Error> {
    loop {
        let msg = receive chan_cap;
        match msg {
            Call(req) => {
                let (resp, new_state) = (ops.handle_call)(req, state)?;
                send caller with resp;
                state = new_state;
            }
        }
    }
}
```

**Type theory:** Manual dictionary passing вҖ” the caller bundles the implementations (the dictionary) and passes it explicitly. This achieves ad-hoc polymorphism at runtime.

**Problems:**
- Runtime overhead of indirect calls
- No inlining/optimization across the boundary
- Verbose at call site

---

**Option 2: Monomorphization (Specialization)**

```ash
// Generic module parameter
module GenServer<M: Handler> {
    fn loop(state: M::State) -> Result<(), Error> {
        let msg = receive chan_cap;
        let (resp, new_state) = M::handle_call(msg, state)?;
        // ...
    }
}

// Concrete implementation
module CounterHandler: Handler {
    type State = Counter;
    type Request = CounterRequest;
    type Response = CounterResponse;
    
    fn handle_call(req, state) -> ... { ... }
}

// Usage - generates specialized code
GenServer<CounterHandler>::start(init_args);
```

**Type theory:** Compile-time specialization. The compiler generates a distinct concrete implementation for each instantiation. Monomorphization is an optimization that eliminates the runtime dictionary.

**Problems:**
- Requires module-level generics (does Ash support this?)
- Each instantiation creates new code (binary bloat)
- Can't have heterogeneous collections of GenServers (different monomorphized types)

---

**Option 3: Language Feature - Type Classes (Not Currently Available)**

```ash
// Hypothetical type class
trait Handler {
    type State;
    type Request;
    type Response;
    
    fn handle_call(req: Request, state: State) -> (Response, State);
}

// Generic code uses implicit dictionary
fn gen_server_loop<H: Handler>(state: H::State) {
    // Compiler passes dictionary implicitly
}
```

**Status:** Not available in Ash. Would require language extension. Type classes provide ad-hoc polymorphism with implicit dictionary passing.

---

**Option 4: Existential Quantification**

```ash
// Hide the concrete handler type
fn spawn_gen_server(handler: exists H. H where H: Handler) -> Address {
    // Can only use Handler methods, don't know concrete type
}

// Or via trait objects (if Ash had them)
fn spawn_gen_server(handler: Box<dyn Handler>) -> Address
```

**Type theory:** Existential quantification hides the concrete type. You know "there exists some type H that implements Handler" but you don't know which H at compile time.

**Use case:** Heterogeneous collections. Store different GenServer types in the same list.

**Problems:**
- Different use case from ad-hoc polymorphism
- Still requires some form of dispatch (vtable or dictionary)
- Not clear if this solves the generic/concrete split for OTP

---

**Option 5: Effect-Based Inversion**

```ash
// Instead of callbacks, use effects
workflow gen_server<State>(handler_cap: HandlerCap<State>) {
    loop {
        effect HandleRequest<State> { /* ... */ };
    }
}
```

**Type theory:** Capabilities as witnesses. The handler capability embodies the concrete implementation.

**Problems:**
- Unclear how this maps to Erlang's callback model
- May require significant restructuring
- Effect handlers vs callback dispatch semantics differ

---

**Option 6: Macro/Template System**

```ash
// Compile-time code generation
gen_server! CounterServer {
    state: Counter,
    handle_call: |req, state| { ... }
}
```

**Type theory:** Staging вҖ” generate code at compile time, then typecheck.

**Key Question:** Is this a **surface language** feature (special syntax) or a **library** feature (proc-macro style)?

If surface language:
- Full type safety, IDE support
- Language complexity increases
- Requires parser/AST changes

If library (macro):
- More flexible, user-extensible
- Error messages may suffer
- Debugger sees generated code
- Hygiene issues

---

### 9.4 The Fundamental Uncertainty

**The Core Problem:**
Erlang achieves the generic/concrete split via dynamic dispatch (module atoms resolved at runtime). Without type classes, Ash must choose between:

1. **Explicit dictionaries** вҖ” verbose, runtime overhead
2. **Monomorphization** вҖ” code bloat, no heterogeneous collections
3. **Language extension** вҖ” type classes, significant work

**Separate Question:** Do we need existentials?
- For heterogeneous supervision trees: possibly yes
- For the basic OTP pattern: no, ad-hoc polymorphism suffices

**Current Blockers:**
1. Ash lacks type classes вҖ” the natural mechanism for ad-hoc polymorphism
2. Unknown: Can Ash do module-level generics? (monomorphization approach)
3. Unknown: Can Ash express dictionaries cleanly? (explicit approach)
4. Unknown: Does Ash have/want macros/templates?

**Status:** Uncertain. Ad-hoc polymorphism is the goal; dictionaries and monomorphization are two ways to get there, each with trade-offs.

See OTP-002 for exploration of design alternatives.
